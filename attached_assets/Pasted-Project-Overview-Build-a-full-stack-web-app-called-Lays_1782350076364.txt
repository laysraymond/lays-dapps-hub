Project Overview

Build a full-stack web app called "Lays Dapps Hub" — a community-run directory of dApps, tools, and games built around the Ritual (ritual.net) AI/crypto ecosystem. These dapps have already been reviewed and approved by Ritual community moderators. This is a community project, not an official Ritual Foundation product. Display "Community Hub — not an official Ritual Foundation product" clearly in the header and footer at all times.

Core purpose: help the Ritual community discover dApps, track which dApps are most clicked through this site specifically, flag dApps by safety status, and require a small testnet-token access fee per browser session to support ongoing maintenance and discourage spam/bot traffic.

Tech stack


Framework: Next.js (App Router) with TypeScript
Styling: Tailwind CSS
ORM: use Prisma specifically — do not use Drizzle or any other ORM. This is a hard requirement, not a default suggestion.
Database: PostgreSQL hosted on Supabase, from the very start — do NOT use SQLite at any point, not even temporarily for local dev. This avoids any future SQLite-to-Postgres migration, since this app will be deployed to Vercel later (ephemeral filesystem, incompatible with file-based SQLite).
Connection strings will be provided as Replit Secrets under these exact names (Replit's runtime reserves the plain name DATABASE_URL for its own built-in database integration, so these custom names must be used instead): SUPABASE_DATABASE_URL (pooled connection, port 6543, via PgBouncer — used for the app's normal runtime queries) and SUPABASE_DIRECT_URL (direct connection, port 5432 — used only by Prisma when running migrations). Set up schema.prisma like this:


prisma  datasource db {
    provider  = "postgresql"
    url       = env("SUPABASE_DATABASE_URL")
    directUrl = env("SUPABASE_DIRECT_URL")
  }


Wallet connection: wagmi + viem, with EIP-6963 multi-wallet discovery (supports MetaMask, OKX Wallet, Rabby, and any other EIP-6963-compliant injected wallet — detect what's installed and show a connector button per wallet, no hardcoded single connector)
No traditional auth system (no NextAuth, no password login anywhere in the app) — admin access and regular access are both gated by wallet address / wallet connection only
Must run on Replit out of the box (bind to 0.0.0.0 and process.env.PORT) and must also be deployable to Vercel later without changes (avoid anything that assumes a persistent local filesystem)

Pages and routes


Home (/) — header with: real Ritual logo at /public/logo.png (I will add this file myself, do not generate/modify it), brand text "Lays Dapps Hub" next to it, "Community Hub — unofficial" label, nav (Home, Leaderboard, Submit, Proof Log, About), small official-link icons (Website/X/Discord/GitHub), and once connected: truncated wallet address + Disconnect button. Hero section. Stats bar (4 cards: total approved dapps, verified count, clicks today, reported count). Weekly leaderboard (top 10 by clicks). Full browsable grid with category filter + search. Status badges (verified=green, unverified=amber, reported=red) always shown with text label, never color alone.
Redirect handler (/go/[id]) — no UI. Increment clickCount, insert ClickLog row, 302 redirect to the real url. Every "Visit" button site-wide must use this route, no direct external links anywhere else.
Leaderboard (/leaderboard) — full sortable leaderboard, toggle "this week" (from ClickLog) vs "all time" (clickCount).
Submit dapp (/submit) — form: name, url, creatorHandle, category, short description (max 200 chars). New submissions: approved: false, status: 'unverified', needsLinkRecheck: false. Validate https url, required name, no duplicate url (case-insensitive).
Proof log (/proof-log) — public, paginated, most recent 200 ClickLog entries (dapp name + relative time only).
About (/about) — public, NOT gated by wallet or fee. Explains the project is community-run/unofficial, what status badges mean, the verification checklist, how to report a dapp, and explains the one-time-per-session access fee before visitors connect.
Admin panel (/admin) — gated by wallet address, not password:

Not connected → show the same multi-wallet connect gate used elsewhere (no fee required here).
Connected but address ≠ ADMIN_WALLET_ADDRESS (case-insensitive compare) → show "This wallet does not have admin access", no admin data exposed.
Connected and address matches → full admin panel: pending submissions (approve/reject), all-dapps list with status/flag controls, reports list with resolve action. Re-verify the admin address server-side in every mutating API route too, never trust client-side checks alone.
Admin access is exempt from the access fee entirely.





Wallet connection + access fee flow (applies to /, /leaderboard, /submit, /proof-log — NOT /about or /admin)


Connect wallet — detect installed wallets via EIP-6963 and show one button per detected wallet (e.g. "Connect with MetaMask", "Connect with OKX Wallet", "Connect with Rabby Wallet"). If only one is detected, show a single "Connect Wallet" button. If none detected, show install links for MetaMask (https://metamask.io/download), OKX Wallet (https://www.okx.com/web3), and Rabby (https://rabby.io).
Check chain — if connected but chainId !== 1979, show "Switch to Ritual Chain" using wagmi's switchChain; this triggers the wallet's add-network flow automatically if Ritual Chain isn't added yet.
Pay the per-session access fee — once connected and on the right chain, check (via a server API route + an httpOnly, no-Max-Age session cookie) whether this browser session has already paid. If not, show:

"This session requires a 0.001 RITUAL access fee (testnet token, no real monetary value) — this is an anti-spam step that also helps support ongoing updates to the dapp directory. You'll need to pay again next time you visit or reconnect."
with a "Pay 0.001 RITUAL" button that uses useSendTransaction (to: FEE_RECEIVER_ADDRESS, value: parseEther("0.001")). Never auto-trigger or auto-approve this — the wallet's own confirmation popup must appear every time, exactly as wallets are designed to require.




After the transaction is sent, wait for confirmation (useWaitForTransactionReceipt), then call a server API route that independently re-verifies on-chain (via https://rpc.ritualfoundation.org) that the tx is confirmed, to matches FEE_RECEIVER_ADDRESS, and value >= parseEther("0.001") — only then set the session cookie and log the payment to PaidAccess. Never trust a client-side "payment succeeded" claim alone.
Session boundary: the session cookie has no Max-Age/Expires (true browser-session cookie). It clears when the browser fully closes. On explicit "Disconnect" click, also clear/invalidate the session cookie via an API call, so reconnecting in the same tab asks for payment again too. Refreshing or navigating between pages within the same open session does NOT ask for payment again.
